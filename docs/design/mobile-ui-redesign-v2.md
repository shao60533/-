# Mobile UI Redesign v2: 信息组织与视觉原型方案

| 项 | 值 |
|---|---|
| Feature | `mobile-ui-redesign-v2` |
| 日期 | 2026-05-09 |
| 范围 | 移动端 IA / UX / UI 方案，不改产品功能、不改技术架构 |
| Demo | [../../demo_mobile_ui_v3.html](../../demo_mobile_ui_v3.html) |

## 1. 结论

当前移动端已经做过一轮工程适配：底部导航、断点、触控尺寸、表格转卡片、横滑 tabs 等基本可用性问题已经被补过。现在“不满意”的核心不再是某个控件溢出，而是移动端仍在沿用桌面端的信息组织：指标、图表、任务、分析、选股、持仓被平铺出来，用户在 375px 屏幕上要自己拼出“现在发生了什么、风险在哪里、下一步做什么”。

v2 的设计方向是：

1. 手机端只回答三个问题：`发生了什么`、`我该看哪里`、`下一步做什么`。
2. 顶层导航保持 4+More，不新增功能，只重排功能入口。
3. 页面首屏从“模块堆叠”改成“决策摘要 + 待处理队列 + 主要动作”。
4. 复杂内容使用渐进展开：摘要卡先给结论，详情、原文、完整表格留在二级 sheet / tab / collapse。
5. 交易类视觉必须高密度但克制：减少大面积装饰，保留数字、趋势、状态、风险优先级。

## 2. 调研依据

- Apple HIG / Tab Bars：iPhone 上 tab bar 适合承载最常用的 app section，复杂 app 可让用户选择或把更多项目放到侧栏/更多入口。参考：<https://developer.apple.com/design/human-interface-guidelines/tab-bars/>
- Material Design / Bottom navigation：底部导航用于移动端 3-5 个同级顶层目的地，超过 5 个应进入替代导航；文字标签要短，避免换行或截断。参考：<https://m1.material.io/components/bottom-navigation.html>
- Android Developers / Layout and navigation：紧凑屏上推荐使用用户熟悉的 navigation bar；drawer 能容纳更多目的地，但在小屏不如底部导航易触达。参考：<https://developer.android.com/design/ui/mobile/guides/layout-and-content/layout-and-nav-patterns>
- Robinhood 导航演进：移动交易产品把 portfolio/watchlist/search/statements/history 这类高频入口放入底部 tab，以降低跨功能跳转摩擦。参考：<https://robinhood.com/us/en/newsroom/a-new-way-to-navigate-robinhood/>
- StockBrokers 2026 Robinhood review：成熟移动交易 app 的关键不是简单，而是在小屏里提供可理解的图表、AI digest、watchlist、alerts，同时避免数据密度过低导致反复切换。参考：<https://www.stockbrokers.com/review/robinhood>
- Apple UI Design Tips：触控控件建议至少 44pt x 44pt。参考：<https://developer.apple.com/design/tips/>

## 3. 现状诊断

### 3.1 已经做对的部分

- `MobileTabbar` 已经采用“仪表盘 / 分析 / 选股 / 持仓 / 更多”的 4+More 结构，方向正确。
- `grid-collapse-mobile`、`mobile-card-header`、`mobile-tabs-scroll`、`mobile-scroll-row` 解决了一批窄屏溢出问题。
- Portfolio / Screener 已有移动卡片 fallback，不是裸表格硬塞手机。
- React island 架构适合渐进式重做页面，不需要推翻 Flask / Vite / API。

### 3.2 主要问题

| 问题 | 当前表现 | 用户影响 |
|---|---|---|
| 信息优先级像桌面 | Dashboard 先堆统计卡、图表、仓位、任务、快捷操作 | 手机首屏没有清楚的“今日重点” |
| 操作与状态分离 | 分析、选股、纸面交易、任务中心在不同页面 | 用户难以判断动作是否已进入队列、是否完成 |
| 复杂配置前置 | Screener V3 直接暴露大师列表、模式、候选数 | 新用户先处理配置，而不是先选目标 |
| 详情页过早展开 | Analysis 详情同时展示 K 线、quick info、pipeline、7 tabs、原文 | 结论不够像“可执行摘要” |
| 更多入口像系统菜单 | More sheet 以功能列表呈现 | 高频辅助入口和低频系统入口混在一起 |
| 视觉层级偏卡片堆叠 | 多个卡片等权排列 | 风险、信号、行动建议没有足够权重 |

## 4. 第一性原理

移动端股票/AI 交易系统的用户不是来“浏览所有数据”的，而是在碎片时间做三类判断：

1. 账户是否安全：亏损、预警、仓位集中、任务失败。
2. 有没有新机会：AI 分析、选股结果、观察列表变化。
3. 是否需要行动：加观察、发起分析、按建议纸面交易、卖出/买入记录。

所以移动端的信息组织应该从“功能模块”转向“决策状态机”：

`监控 -> 发现 -> 分析 -> 决策 -> 追踪`

对应到现有功能，不需要改能力：

| 状态 | 使用现有功能 | 移动端呈现 |
|---|---|---|
| 监控 | Dashboard / Alerts / Portfolio | 首页顶层健康摘要 |
| 发现 | Screener V3 / Watchlist | 机会队列和筛选入口 |
| 分析 | Analysis run / History | 分析命令栏和结果 inbox |
| 决策 | Advice / Trade decision | 结论卡 + CTA |
| 追踪 | Paper trade / Tasks | 追踪状态、订单、任务进度 |

## 5. 新信息架构

### 5.1 底部导航

保留当前 4+More，但重新定义语义：

| 当前 | v2 文案 | 内容 |
|---|---|---|
| 仪表盘 | 首页 | 账户健康、今日队列、风险、运行中任务 |
| 分析 | 分析 | ticker 命令栏、分析 inbox、详情摘要 |
| 选股 | 发现 | 策略预设、筛选结果、候选比较 |
| 持仓 | 持仓 | 仓位、交易记录、预警关联 |
| 更多 | 更多 | 报告、回测、纸面交易、任务、设置 |

命名原因：`发现` 比 `选股` 更符合用户任务，里面仍使用现有 Screener V3 功能；如果担心改文案影响老用户，可第一期保持“选股”，但页面内部用“发现机会”组织。

### 5.2 首页首屏

目标：用户打开手机 5 秒内知道是否需要处理事情。

首屏结构：

1. 顶部状态条：市场状态、模型、同步状态。
2. 账户健康摘要：总资产、今日/总盈亏、风险灯号。
3. 决策队列：`需要复核`、`新机会`、`运行中` 三类。
4. 主要动作：分析一只股票 / 开始发现 / 添加交易。

图表下移到第一屏后半，不再压过风险与行动。

### 5.3 分析页

目标：把“发起分析”和“读懂结论”放在同一个流里。

移动端结构：

1. Sticky 命令栏：Ticker 输入 + 深度 segmented control + 发送。
2. 运行态：紧凑进度条，点开看 Pipeline DAG。
3. 完成态：结论卡优先展示 `Buy/Sell/Hold`、置信度、风险、AI 摘要。
4. 详情 tab：概览 / 技术 / 基本面 / 新闻 / 辩论 / 风险 / 原文。
5. CTA：加入观察、按建议纸面交易、生成报告。

### 5.4 发现页 / Screener V3

目标：先选“投资任务”，再展开大师配置。

移动端结构：

1. 三个策略预设：质量价值、成长动量、宏观防守。
2. 高级配置折叠：大师、多空辩论、候选数、成本预估。
3. 结果榜单卡片化：代码、综合分、共识、投票条、风险提示、分析 CTA。
4. Top 5 圆桌辩论折叠在每只股票详情里，不在结果页上方抢首屏。

### 5.5 持仓页

目标：不仅看仓位，还能看到“哪只需要处理”。

移动端结构：

1. 持仓健康条：集中度、最大亏损、活跃预警。
2. 持仓列表按风险/变动排序，可切换市值排序。
3. 每个持仓卡展示：价格、盈亏、仓位、最近 AI 信号、预警状态。
4. 买入/卖出保留现有弹窗能力，入口改为上下文按钮。

### 5.6 更多页

目标：把低频系统能力收纳清楚。

分组：

- 复盘：报告中心、策略回测、纸面交易。
- 运营：任务中心、预警中心。
- 系统：模型设置、账号设置。

不要只写“导航到其他页面”，更多页本身也要有信息气味，比如任务失败数、最近报告时间。

## 6. UI 语言

### 6.1 视觉层级

- 一级：决策结果、风险状态、当前任务。
- 二级：趋势图、持仓摘要、候选排名。
- 三级：原始报告、历史记录、参数明细。

### 6.2 组件模式

| 模式 | 用途 | 规范 |
|---|---|---|
| Decision Card | AI 分析结论、候选股票、风险项 | 左侧 ticker/状态，右侧信号；中间一段摘要；底部 CTA |
| Health Strip | 首页/持仓健康 | 3-4 个指标横向滚动或 2x2 |
| Command Bar | 发起分析/筛选/添加交易 | 单行输入 + 主 CTA，滚动后 sticky |
| Queue | 任务和待处理项 | 按优先级排序，状态颜色只表达风险 |
| Detail Sheet | 大师解释、完整投票、原文报告 | 默认收起，用户主动打开 |

### 6.3 色彩

继续沿用现有暗色金融终端风格，但减少单一蓝紫感：

- 背景：深中性灰，不要过多渐变。
- Buy：绿色；Sell：红色；Hold/Watch：黄色或中性。
- 信息：蓝色只用于可点击/当前导航。
- 风险：红/黄要克制使用，避免全屏报警疲劳。

## 7. 分阶段落地

### Phase 1：不改功能的 UX 壳调整，约 1-2 天

- 调整 mobile tab 文案和 More sheet 分组。
- Dashboard 移动端重排：健康摘要 + 决策队列前置。
- Analysis 列表页把“发起分析”命令栏前置。
- Screener 移动端把高级配置折叠。
- 增加 `DecisionCard`、`HealthStrip`、`MobileCommandBar`、`PriorityQueue` 共享组件。

### Phase 2：关键页面细化，约 2-3 天

- Analysis detail 移动端改成“结论摘要 -> 图表 -> tabs -> 原文”。
- Screener result 卡片增强：投票条、共识、风险、CTA。
- Portfolio 卡片接入最近 AI signal / alert 状态。
- More sheet 增加任务/报告/预警状态预览。

### Phase 3：验证与回归，约 1 天

- 375 / 390 / 430 / 768px 视口截图。
- body 无横向滚动。
- 所有主要 touch target >= 44px。
- 首页首屏能看到总资产、风险、至少一个主要动作。
- Analysis 发起后同页能看到运行态，不跳丢上下文。

## 8. 验收标准

| 指标 | 标准 |
|---|---|
| 首屏可理解性 | 375px 下首页无需滚动可看到资产、风险、主要动作 |
| 导航 | 底部最多 5 个入口；More 分组清楚 |
| 行动闭环 | 分析/选股结果卡上能直接进入下一步 |
| 信息密度 | 卡片默认只展示 P0 信息，详情折叠 |
| 无功能变化 | API、路由、任务流、数据结构不变 |
| 无架构变化 | React island / Flask shell / Tailwind / shadcn 风格保持 |
| 可访问性 | 主按钮和 icon button 点击区域 >= 44px |
