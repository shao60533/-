# 股票辅助决策系统 V2.0 — 产品需求文档 (PRD)

> **版本**: 2.0  
> **日期**: 2026-04-12  
> **状态**: 草稿 — 待评审  
> **作者**: AI Product Spec (基于现有代码库分析)

---

## 一、问题陈述（Problem Statement）

当前系统拥有完整的后端能力（多源数据获取、三层智能选股、7 Agent AI 分析、持仓管理、预警监控、回测引擎、报告生成、Telegram/Email 通知），但 **Web 端几乎不可用**：9 个页面中 8 个为空壳，前端 JS 90% 为占位函数，后端 API 仅覆盖 3 组端点（dashboard / portfolio / price），用户只能通过 CLI 使用系统。

**谁受影响**：需要随时查看持仓、执行分析、接收预警的个人投资者（即系统作者本人），使用场景以手机为主。

**不解决的代价**：系统的 AI 分析、三层选股、策略回测等核心能力被锁在命令行里，无法在手机上使用，无法在非技术场景下快速决策。一个功能完整但界面残缺的产品 = 一个不可用的产品。

## 二、目标用户画像

**主要用户**：系统作者本人 — 有编程能力的个人投资者

| 特征 | 描述 |
|------|------|
| 技术水平 | 高。能配置 API Key、理解 CLI、能部署云服务 |
| 交易市场 | 美股（IB/Polygon）+ A 股（AkShare） |
| 使用设备 | **手机为主**（通过 Railway 云端部署后远程访问），偶尔桌面 |
| 核心诉求 | 快速查看 AI 分析结论 → 决定是否操作 → 手动在券商下单 |
| 使用频率 | 盘前/盘后各 1-2 次，盘中收到预警时临时查看 |
| 痛点 | CLI 无法在手机使用；缺乏可视化；分析结果没有持久化界面 |

**非目标用户**：不服务多人协作、机构交易、自动化下单场景。

## 三、目标（Goals）

| # | 目标 | 衡量方式 |
|---|------|----------|
| G1 | **Web 端功能完整可用** — 9 个页面全部可操作，CLI 能做的事 Web 都能做 | 功能覆盖率 = Web 功能数 / CLI 功能数 ≥ 95% |
| G2 | **手机端核心操作路径 ≤ 2 步** — 从打开到看到 AI 分析结论不超过 2 次点击 | 核心路径操作步数（分析、查看持仓、查看预警） |
| G3 | **分析结果可追溯** — 历史分析记录持久化，支持回看和时间线对比 | 分析历史记录数 > 0 且可检索 |
| G4 | **实时感** — 价格/盈亏/预警在 Web 端有实时更新反馈 | WebSocket 推送延迟 < 3s |
| G5 | **零运维可部署** — 一键部署到 Railway，无需额外配置即可通过手机访问 | `railway up` 后 Web 可访问 |

## 四、非目标（Non-Goals）

| # | 不做什么 | 原因 |
|---|----------|------|
| NG1 | **不做自动化交易（auto-trade）** | 系统定位是"辅助决策"而非"自动执行"。下单仍在券商 App 完成。混合决策与执行会带来不可逆的财务风险。 |
| NG2 | **不做多用户/权限系统** | 单人使用，增加 auth 层只是工程负担。如果部署到公网，在 Railway 层面加 Basic Auth 即可。 |
| NG3 | **不做原生 App（iOS/Android）** | PWA 足以覆盖手机场景。原生 App 的开发和维护成本远超单人项目承受范围。 |
| NG4 | **不做实时 K 线图/盘中 tick 级行情** | 系统的数据源（yfinance/Polygon 免费层/AkShare）均为延迟数据，做实时 K 线是虚假承诺。聚焦 AI 分析结论而非替代券商行情软件。 |
| NG5 | **不做社交/分享功能** | 单人工具，没有分享需求。 |

## 五、用户故事（User Stories）

### 5.1 仪表盘 — "一眼看全貌"

**US-1.1** 作为投资者，我想打开 Web 就看到总资产、总盈亏、收益率和活跃预警数，这样我不用任何操作就能知道"今天账户怎么样"。

**US-1.2** 作为投资者，我想看到持仓列表的实时盈亏（价格变动时有颜色闪烁），这样我能在盘中快速判断哪些仓位需要关注。

**US-1.3** 作为投资者，我想在仪表盘看到净值曲线（7天/1月/3月/1年），这样我能判断近期趋势是向上还是向下。

**US-1.4** 作为投资者，我想在仪表盘看到仓位分布饼图，这样我能判断是否过于集中在某只股票。

**US-1.5** 作为投资者，我想在仪表盘有"快捷操作"入口（分析、选股、报告），这样不用翻菜单就能直达高频功能。

### 5.2 AI 分析 — "核心决策引擎"

**US-2.1** 作为投资者，我想输入股票代码后一键触发 AI 分析，这样我不需要敲 CLI 命令也能获得多 Agent 分析结果。

**US-2.2** 作为投资者，我想在分析过程中看到每个 Agent 的实时进度（技术面 ✅ → 基本面 ✅ → 情绪 🔄 → …），这样我知道分析进行到哪一步、还需要等多久。

**US-2.3** 作为投资者，我想看到分析结论的三张摘要卡片（信号 BUY/SELL/HOLD、置信度百分比、建议操作），这样我不用读完整报告就能快速做决定。

**US-2.4** 作为投资者，我想通过 Tab 切换查看各维度详细报告（技术面、基本面、情绪、新闻、多空辩论），这样我能深入了解结论背后的推理。

**US-2.5** 作为投资者，我想看到"多空辩论"的双栏对比展示，这样我能同时看到看多和看空的论点，避免确认偏误。

**US-2.6** 作为投资者，我想分析完成后结果自动保存到历史记录，这样我不需要手动保存也能回看。

### 5.3 分析记录 — "追溯与对比"

**US-3.1** 作为投资者，我想按股票代码搜索历史分析记录，这样我能快速找到某只股票过去的分析结论。

**US-3.2** 作为投资者，我想看到同一只股票的信号变化时间线（04-12 BUY → 04-05 HOLD → 03-28 BUY），这样我能判断 AI 判断的稳定性和趋势方向。

**US-3.3** 作为投资者，我想选择两条记录做并排对比，这样我能看到各维度在不同时间的差异。

### 5.4 智能选股 — "发现机会"

**US-4.1** 作为投资者，我想选择市场（美股/A 股/全部）和策略（成长/价值/动量/低波动）后一键开始选股，这样我不用手动配置复杂的筛选条件。

**US-4.2** 作为投资者，我想看到三层筛选的漏斗可视化（IB 150 只 → finviz 28 只 → AI 精选 8 只），这样我能理解筛选过程并信任最终结果。

**US-4.3** 作为投资者，我想在选股结果列表中直接点击某只股票跳转到 AI 分析页，这样从"发现"到"深入研究"只需一步。

### 5.5 持仓管理 — "记录与追踪"

**US-5.1** 作为投资者，我想在 Web 端通过表单记录买入/卖出操作（股票、数量、价格、日期、备注），这样我不用打开终端也能维护持仓记录。

**US-5.2** 作为投资者，我想看到当前持仓表（含实时市值、盈亏金额、收益率百分比），表格数据右对齐、数字等宽，这样数据一目了然。

**US-5.3** 作为投资者，我想看到仓位分布饼图和盈亏汇总数据，这样我能评估整体组合健康度。

**US-5.4** 作为投资者，我想在持仓表中每行有"分析"和"卖出"快捷按钮，这样我能直接对某只持仓执行操作而不用切换页面再输入代码。

**US-5.5** 作为投资者，我想查看交易记录（支持按股票筛选），这样我能复盘自己的买卖时机。

### 5.6 预警中心 — "不错过关键信号"

**US-6.1** 作为投资者，我想在 Web 端创建预警规则（价格突破、跌幅超限、放量、止损、止盈），这样我不用记 CLI 命令语法。

**US-6.2** 作为投资者，我想看到每条规则的实时状态（监控中🟢 / 已触发🔴）和当前价格距触发阈值的距离，这样我能提前预判。

**US-6.3** 作为投资者，我想用预设模板（止损 -8%、止盈 +15%、突破前高）快速创建常用规则，这样不用每次手动填参数。

**US-6.4** 作为投资者，我想预警触发时在 Web 端收到实时通知（Toast 弹窗 + 铃铛角标），这样即使没看 Telegram 也不会错过。

**US-6.5** 作为投资者，我想查看预警触发历史（何时触发、当时价格），这样我能复盘预警的有效性。

### 5.7 报告中心 — "定期总结"

**US-7.1** 作为投资者，我想在 Web 端选择报告类型（日报/周报/月报/个股深度）后一键生成报告，这样我不用跑 CLI 命令。

**US-7.2** 作为投资者，我想在页面内阅读渲染好的 Markdown 报告（非原始文本），这样报告有良好的排版和可读性。

**US-7.3** 作为投资者，我想一键复制报告文本或发送到 Telegram，这样我能方便地保存或转发。

### 5.8 策略回测 — "验证想法"

**US-8.1** 作为投资者，我想选择股票、策略（SMA 交叉/RSI 均值回归/买入持有）、时间范围和初始资金后开始回测，这样我能验证某个策略在历史数据上的表现。

**US-8.2** 作为投资者，我想看到回测净值曲线（叠加基准对比），这样我能直观看到策略是否跑赢大盘。

**US-8.3** 作为投资者，我想看到关键绩效指标（总收益、最大回撤、胜率、交易次数、年化收益），这样我能量化评估策略质量。

**US-8.4** 作为投资者，我想查看每笔交易的明细（入场/出场日期、价格、盈亏、持仓天数），这样我能分析策略在哪些行情下表现好或差。

### 5.9 设置 — "管理配置"

**US-9.1** 作为投资者，我想在 Web 端配置 API Key（Gemini、Polygon、Qwen），这样我不用编辑 YAML 文件。

**US-9.2** 作为投资者，我想配置通知渠道（Telegram Bot Token/Chat ID、Email SMTP），这样预警能推送到我的手机。

**US-9.3** 作为投资者，我想启停定时任务（每日报告、每周报告、预警检查间隔），这样我能控制系统的自动化行为。

**US-9.4** 作为投资者，我想 API Key 以脱敏形式显示（`****abc1`），只在"修改"时才输入完整新值，这样避免泄露。

### 5.10 全局体验

**US-10.1** 作为投资者，我想用全局搜索栏快速找到任何股票相关的信息（持仓、分析记录、预警），这样我不用逐个页面查找。

**US-10.2** 作为投资者，我想在数据加载时看到骨架屏（而不是空白或旋转菊花），这样我知道内容即将呈现且页面没有卡死。

**US-10.3** 作为投资者，我想所有操作（分析完成、买入成功、预警触发）都有 Toast 通知反馈，这样我确认操作已执行。

**US-10.4** 作为投资者，我想在手机上使用时表格自动变成卡片列表、模态框变成全屏抽屉，这样手指能精准操作而不是在小表格里戳来戳去。

## 七、后端 API 完整规格

以下列出所有需要补全的 API 端点。**已有端点**标注 ✅，**需新增**标注 🆕。

### 7.1 仪表盘

| 端点 | 方法 | 状态 | 说明 |
|------|------|------|------|

### 7.2 AI 分析

| 端点 | 方法 | 状态 | 请求体 | 响应 |
|------|------|------|--------|------|
| `/api/analysis/run` | POST | 🆕 | `{ticker, date?}` | `{task_id}` → 通过 WebSocket `analysis_step` 推送进度 → 完成后 `analysis_complete` 推送完整结果 |
| `/api/analysis/history` | GET | 🆕 | Query: `?ticker=&page=&limit=` | `{items: [{id, ticker, signal, confidence, date, summary}], total}` |
| `/api/analysis/<id>` | GET | 🆕 | — | 完整分析结果（signal, reports, debate, advice） |
| `/api/analysis/<id>` | DELETE | 🆕 | — | `{"deleted": true}` |

| 端点 | 方法 | 状态 | 请求体 | 响应 |

## 八、前端改造规格

### 8.1 需修改的文件

| 文件 | 改动性质 | 估算行数 |
|------|----------|----------|
| `index.html` | **重写** — 补全 8 个空页面的完整 HTML 结构 | ~800 行（现 210 行） |
| `app.js` | **重写** — 占位函数全部替换为真实 API 调用 + ECharts 渲染 + WebSocket 事件处理 | ~1200 行（现 192 行） |
| `style.css` | **增量** — 新增组件样式（保留现有暗色主题基础） | +500 行（现 700 行） |

### 8.2 每个页面必须包含的状态

所有 9 个页面在实现时必须覆盖以下 4 种状态：

| 状态 | 表现 |
|------|------|
| **加载中** | 骨架屏或 spinner + "加载中…" 提示（P0 用 spinner，P2 升级为骨架屏） |
| **有数据** | 正常展示内容 |
| **空数据** | 大图标（64px, 低透明度）+ 文字提示 + 引导操作按钮（如"添加第一笔持仓"） |
| **错误** | 红色警告图标 + 错误消息 + [重试] 按钮 |

### 8.3 前端技术栈（保持现有，不引入新框架）

| 依赖 | 版本 | 用途 | 引入方式 |
|------|------|------|----------|
| Bootstrap 5.3 | CDN | 网格 + 基础组件 | 已有 |
| Font Awesome 6.5 | CDN | 图标系统 | 已有 |
| ECharts 5.x | CDN | 数据可视化 | 已有（未初始化） |
| Socket.IO 4.7 | CDN | WebSocket 通信 | 已有（未接事件） |
| marked.js | CDN | Markdown → HTML 渲染 | **新增** |

**不引入 React/Vue/Svelte** — 系统为单页应用但页面间交互简单，原生 JS + DOM 操作足够。引入框架会增加构建步骤和部署复杂度，违反"零运维"原则。
